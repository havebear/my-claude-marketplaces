# 项目分析报告

> 分析时间：2026-03-13

## 项目概述

**my-claude-marketplaces** 是一个 Claude Code 插件集合，通过 Marketplace 机制分发，为开发者提供项目规范管理、文件保护、AI 辅助调试、知识库管理和智能通知等功能。

- Marketplace 版本：3.13.0
- 插件数量：5 个
- 技术栈：Markdown + JSON（插件定义）、JavaScript/Node.js（Hook 脚本）

## 项目作用与实现原理

### 项目作用

这是一个 **Claude Code 插件集合（Marketplace）**，本质上是给 Claude Code 这个 AI 编程助手"装插件"，扩展它的能力。

用一句话概括：**让 Claude Code 更好用、更安全、更智能。**

### 实现原理

Claude Code 有一套插件机制，支持以下 4 种扩展点：

**1. Commands（斜杠命令）**

用 Markdown 文件定义，用户输入 `/ps-plan` 等命令时，Claude 读取该 `.md` 文件作为提示词，按照文件中的指令执行任务。

```
/ps-plan → 读取 commands/ps-plan.md → Claude 按指令进行多轮规划对话
```

**2. Skills（技能）**

同样是 Markdown 文件，但通过自然语言触发（不需要斜杠命令）。Claude 检测到用户意图匹配时自动加载对应技能的上下文。

**3. Hooks（钩子）**

在特定事件发生时自动执行 Node.js 脚本，类似中间件：

```
用户让 Claude 编辑文件
    → PreToolUse Hook 触发
    → 执行 validate-edit.js
    → 判断路径是否在项目内
    → 允许 or 拦截并询问用户
```

**4. Agents（子 Agent）**

Markdown 定义的专用 AI 实例，可被主 Claude 调用来完成特定子任务（如 `kb-researcher` 专门负责检索知识库）。

### 整体运行流程

```
开发者安装 Marketplace
    ↓
Claude Code 加载插件配置（plugin.json / marketplace.json）
    ↓
运行时：Commands / Skills / Hooks / Agents 按需触发
    ↓
增强 Claude Code 的规划、保护、通知、知识管理能力
```

核心思路：**用 Markdown 描述"Claude 应该怎么做"，用 JS 脚本处理需要程序逻辑的部分**，两者结合实现功能扩展。

### Marketplace 与插件的层级关系

my-claude-marketplaces 本质上是一个 **Marketplace**，相当于"插件商店"，内部管理多个子插件。

**层级结构（扁平，非树状）：**

```
Claude Code
├── Marketplace A（my-claude-marketplaces）
│   ├── plugin: project-spec
│   ├── plugin: file-guard
│   ├── plugin: notify
│   ├── plugin: codex
│   └── plugin: knowledge-base
├── Marketplace B（另一个 marketplace）
│   ├── plugin: xxx
│   └── plugin: yyy
└── ...
```

**重要限制**：
- Marketplace **不能嵌套** Marketplace，`marketplace.json` 的 `plugins[]` 只能指向本地插件目录
- 多个 Marketplace 之间是**平级关系**，各自独立管理
- Plugin **不能包含子 Plugin**，`plugin.json` 只注册 commands / skills / hooks / agents
- 如需逻辑分组，只能通过目录命名约定或文档来组织，配置层面不支持树状结构

## Claude Code 插件系统

### 系统概述

Claude Code 插件系统是一套扩展框架，允许开发者通过模块化组件为 Claude Code 添加自定义能力。插件以 Markdown + JSON 为主要载体，轻量且无需编译。

### 四种组件类型

**Commands（斜杠命令）**
- 用 `.md` 文件定义，用户通过 `/command-name` 显式触发
- 文件内容即为 Claude 的执行指令
- 例：`/ps-plan` → 读取 `ps-plan.md` → Claude 按指令进行多轮规划

**Skills（技能）**
- 同样是 `.md` 文件，但通过自然语言模式匹配自动触发
- 无需用户输入斜杠命令，Claude 识别意图后主动加载
- 例：用户说"帮我制定计划" → 自动激活 `planning` skill

**Hooks（钩子）**
- 事件驱动，在特定系统事件发生时自动执行
- 支持 Node.js 脚本或 AI 评估来决定是否拦截操作
- 常见事件：`PreToolUse`、`UserPromptSubmit`、`Stop`、`Notification`

**Agents（子 Agent）**
- Markdown 定义的专用 AI 实例，可独立执行复杂子任务
- 由主 Claude 调用，拥有独立上下文

### Marketplace 机制

Marketplace 是插件的分发载体，通过 `.claude-plugin/marketplace.json` 定义：

```json
{
  "name": "my-claude-marketplaces",
  "version": "3.13.0",
  "plugins": [
    { "name": "project-spec", "source": "./plugins/project-spec" }
  ]
}
```

安装整个插件集合：

```bash
/plugin marketplace add qqabcv520/my-claude-marketplaces
```

单独安装某个插件：

```bash
/plugin install file-guard@my-claude-marketplaces
```

### 插件加载流程

```
Claude Code 启动
    ↓
读取 marketplace.json → 发现所有插件
    ↓
读取每个插件的 plugin.json → 注册组件
    ↓
Commands 注册为斜杠命令
Skills 注册为自然语言触发器
Hooks 绑定到对应系统事件
Agents 注册为可调用子实例
```

### 核心设计理念

- **Markdown-first**：插件逻辑用自然语言描述，Claude 本身就是执行引擎
- **JS 处理程序逻辑**：需要条件判断、文件操作等场景才用 Node.js 脚本
- **零框架依赖**：除 notify 插件用了 `node-notifier`，其余无外部依赖

## 技术架构

### 插件标准结构

```
plugin-name/
├── .claude-plugin/plugin.json    # 插件元数据与组件注册
├── commands/                     # 命令定义（.md 文件）
├── skills/                       # 技能定义（.md 文件）
├── agents/                       # Agent 定义（.md 文件）
├── hooks/
│   ├── hooks.json                # Hook 配置
│   └── scripts/                  # Hook 实现（JS）
│       └── tests/                # 测试文件
├── lib/                          # 共享库（JS）
└── README.md                     # 插件文档
```

### 编码规范

- 插件名：kebab-case（如 `my-plugin`）
- 命令文件：kebab-case（如 `my-command.md`）
- JSON 配置：2 空格缩进
- 文档语言：中文
- 版本管理：语义化版本（SemVer）

## 插件开发指南

### 最小可用插件结构

只需两个文件即可创建一个可用插件：

```
my-plugin/
├── .claude-plugin/plugin.json
└── commands/hello.md
```

`plugin.json` 最简配置：

```json
{
  "name": "my-plugin",
  "version": "1.0.0",
  "description": "描述",
  "commands": ["./commands"]
}
```

### Command 文件即提示词

命令文件的内容直接决定 Claude 执行该命令时的行为，标准结构：

```markdown
# 命令名称

## 执行指令

当此命令被调用时：
1. 步骤一
2. 步骤二
```

命名规范：文件名与命令名对应（`/my-command` → `my-command.md`），跨插件命令使用 `插件名:命令名` 格式（如 `/notify:init`）。

### Hook 两种类型对比

| 类型 | 适用场景 | 建议 timeout |
|------|---------|-------------|
| `command` | 需要程序逻辑（路径判断、文件读写） | 10s |
| `prompt` | 需要 AI 理解语义（判断意图、注入上下文） | 30s |

`command` 类型通过 stdin 接收参数，stdout 返回 JSON：

```json
{ "decision": "approve" }
// 或
{ "decision": "block", "reason": "原因" }
```

`prompt` 类型可注入系统消息：

```json
{ "decision": "approve", "systemMessage": "注入的上下文" }
```

### Hook 支持的所有事件

| 事件 | 触发时机 | 典型用途 |
|------|---------|---------|
| `PreToolUse` | 工具调用前 | 拦截危险操作 |
| `PostToolUse` | 工具调用后 | 日志、通知 |
| `UserPromptSubmit` | 用户提交 prompt | 注入上下文 |
| `Stop` | AI 停止响应 | 总结、保存 |
| `SubagentStop` | 子 Agent 停止 | 结果检查 |
| `SessionStart` | 会话开始 | 初始化 |
| `PreCompact` | 压缩上下文前 | 保留关键信息 |
| `Notification` | 通知事件 | 桌面通知 |

### Skill 渐进式加载

Skill 目录建议分三层，避免一次性加载过多内容：

```
skills/my-skill/
├── SKILL.md        # 核心内容，始终加载
├── examples/       # 示例，按需引用
└── references/     # 参考资料，按需引用
```

### 发布与安装

推送到 GitHub 后，其他人即可安装：

```bash
/plugin marketplace add your-username/your-repo
```

### 开发建议

- **先做 Command**：最简单，改 Markdown 即可迭代
- **Hook 的 JS 脚本写测试**：调试成本高，TDD 可以节省时间
- **Skill 触发条件要具体**：太模糊会频繁误触发
- **版本号每次改动都要更新**：`plugin.json` 和 `marketplace.json` 同步更新
